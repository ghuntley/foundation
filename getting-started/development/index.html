Development

Building‚Äã

Let's build the example servers to make sure everything is setup correctly. Spoiler: This step is optional, other commands you'll use later like dev or deploy run build implicitly. But bear with us.

fn build api/server

You may notice that Foundation downloaded Go in order to build the server. This is a private copy that is managed by Foundation and kept in a SDK cache, and doesn't interfere with your system. It is part of our dependency management. When the project's dependencies change later, we'll update them automatically for you.

fn build validates configurations for the servers, passed as arguments, and all their services and transitive dependencies. Then it builds all the servers in parallel, invoking language-specific standard build tools (go for Go, vite for Javascript).

Lets try now:

fn build web/server

Foundation lists as a result not just the web server, but also the api server we built earlier. There's a relationship between the servers, and the platform knows that, and it knows to build all transitive dependencies as needed.

Start a Development Session‚Äã

When developing a service or set of services we're caught up in a build, deploy and test cycle. Foundation provides you with development sessions that do that for you:

  • Builds all the required servers.
  • Deploys them locally (or to a remote cluster).
  • Sets up ports forwarding so the servers are available from the local workstation.
  • Starts the Foundation Web UI (as an extra server) for debugging.
  • Watches source files and does rebuild/restart on changes (edit/refresh).

fn dev api/server web/server

Edit the Code and Use Edit/Refresh‚Äã

Our example app has a bug: deleting items (via x button on the right side of an item) doesn't work. Let's see how we can fix it.

Open api/todos/impl.go and uncomment the following lines:

// "Development" User Journey:
// Uncomment next 3 lines.

// if _, err := db.Exec(ctx, "DELETE FROM todos_table WHERE ID = $1;", req.Id); err != nil {
//   return fmt.Errorf("failed to remove todo: %w", err)
// }

You can see that the api/server got rebuilt and redeployed:

[+] 53.9s 14/16 actions completed
 image.publish publish="fn.publish.registry" tag="k3d-registry.nslocal.dev:41000/namespacelabs.dev/examples/todos/ap (925ms)
 => oci.write-image 0.00% ref="k3d-registry.nslocal.dev:41000/namespacelabs.dev/examples/todos/api/server:htsbc3824t (925ms)

Now deleting items works in the UI!

We can also modify the build configuration (for example, add a new dependency), and fn dev will automatically re-deploy the server.

tip

Whereas for Go servers a rebuild is required, for Web servers Foundation supports HMR (Hot Module Replacement) to instantly reload the UI when the source files change.

Try changing the button name in todos/web/ui/src/AddTodoForm.tsx to see it in action!

Use the Debug UI‚Äã

Open the Debug UI (http://127.0.0.1:4001 by default, our check out fn dev's output).

It allows to inspect the endpoints of the running, observe logs, execute commands in the server container, etc.

Send gRPC requests via command line‚Äã

fn dev eventually brings a list of exported services that can be accessed from the workstation via grpcurl by the printed addresses with forwarded ports.

# Install "grpcurl" on the workstation or use the fn-provided one:
fn tools grpcurl -plaintext 127.0.0.1:39737 list
